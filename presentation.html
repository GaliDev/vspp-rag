Section 2 / 4 · Retrieval, step by step VSPP RAG · GenAI Mini-Hackathon

From a question to top-k chunks — in six steps.

The previous slide showed every box. This is just the retrieval path, in plain English.

1
User asks a question
CLI, API call, or eval harness — same entry point.
↓
2
Load the vector index
vectors.npy (2,868 × 384 floats) + chunk_index.jsonl (per-chunk metadata).
↓
3
Optional: metadata router creates filters
route_query() → RetrievalFilters: external_id, category, source, exclude lists. This is where the corpus gets narrowed before any vector math.
↓
4
Convert the question into an embedding vector
Same MiniLM-L6-v2 that embedded the chunks. Output: 384 numbers, length-normalized.
↓
5
Search only chunks that match the filters
Cosine similarity (one BLAS matmul) → score for every surviving chunk. Sort descending.
↓
6
Return top-k chunks with citations
Each hit carries external_id, char_start, char_end, score, and the original text.

Section 2 / 4 · Where the design lives VSPP RAG · GenAI Mini-Hackathon

The router is the whole point.

A naive RAG hits one vector index. We hit metadata first, then a smaller vector space — two layers, same code path.

Layer 1

Cheap regex router

route_query() in src/core/retrieval.py turns query text into a RetrievalFilters object:

  • external_ids — pin to one spec (“IMSC1.1” → w3c-tr-imsc11)
  • categories — Structural/System vs Transport
  • sources — ado_wiki, confluence, …
  • exclude_external_ids — kick out conformance fixtures when the query is normative

Fast, transparent, debuggable. Brittle — needs a new rule per spec family.

Layer 2

Doc-summary router (semantic)

During normalize, a local Qwen2.5-1.5B writes a 3-sentence summary per doc into manifest.metadata.doc_summary.

chunk.py emits a virtual chunk per summary (chunk_kind: doc_summary). At query time, retrieve.py --doc-summary-router does:

  • Stage 1 — cosine over ~131 summary vectors → pick top docs
  • Stage 2 — cosine over text chunks inside those docs only

General — works for new specs with no new code. Cost: one local LLM pass per doc at ingest.

Section 3 / 4 · Tech review — prep stack VSPP RAG · GenAI Mini-Hackathon

What we picked & why — prep side

Component We picked Why Alternative considered
Discovery async collectors + discovery_manifest.json Single source of truth; merge preserves prior ingest state across re-runs (merge_discovery_preserving_ingest). Crawl-on-demand with no manifest — simpler, but every re-run loses ingest state and provenance.
Normalization pypdf · BS4 · docx XML · zip bundles One normalizer per file type; W3C TR HTML and GitHub repo archives avoid lossy PDF rendering entirely. unstructured.io / pdfplumber — better tables, much heavier deps, slower for our mostly-prose specs.
Doc summary Qwen2.5-1.5B-Instruct (local) Offline demo, no API keys, zero data exfil — important because some sources are internal wikis. gpt-4o-mini in the cloud — faster & better summaries, but cost + privacy + offline-demo break.
Chunking paragraph_pack_v1 · 2000 chars / 200 overlap Spec text has hard paragraph structure — packing paragraphs preserves clause boundaries without fixed-token cuts. Semantic chunking (LLM-decided breaks) — better quality, but expensive at 131 docs and harder to debug.
Embeddings sentence-transformers all-MiniLM-L6-v2 · 384-d ~80 MB model, runs on CPU, ~50 ms/query. Good-enough recall on technical English at POC scale. BGE-base-en / E5-base — measurably better quality, 4× larger, slower; would matter past ~50 k chunks.

Section 3 / 4 · Tech review — retrieval stack VSPP RAG · GenAI Mini-Hackathon

What we picked & why — retrieval side

Component We picked Why Alternative considered
Vector store NumPy vectors.npy + JSONL chunk_index.jsonl 2,868 × 384 floats = ~4 MB. Whole index fits in RAM; cosine is a single BLAS matmul (< 1 ms). pgvector / Qdrant / FAISS — necessary past ~50 k chunks for ANN + filter as partition key. Overkill today.
Routing Regex route_query() + doc-summary 2-stage Layer 1 is precision; layer 2 generalizes without writing new rules per spec. Both produce the same RetrievalFilters. Pure LLM-based router — flexible but slow (~1 s/query) and not deterministic enough to unit-test.
Scoring masked cosine on filtered subset One code path for filtered & unfiltered — build_mask → np.where → vectors[idx] @ q. Trivial to read and trace. Hybrid BM25 + dense + reranker — better top-1 quality, three more moving parts to maintain.
Citations (external_id, char_start, char_end) on every chunk Byte-exact provenance — judges & auditors can verify any answer against the original normative source. Page-number citations from PDF — lossy after normalization, doesn’t survive HTML or repo sources.
Evaluation eval_retrieval.py + JSONL queries · modes: baseline / router / summary-router 13 ground-truth queries; we can grade a code change in 9 seconds. Same harness graded the eval on the next slide. LLM-as-judge for faithfulness — needed once we add a final answer step. Not in POC scope.

Why baseline fails

Conformance fixtures drown the spec

The MPEGGroup/FileFormatConformance repo is 1,237 chunks — 44% of the corpus. It’s full of JSON test fixtures with strings like "@Type": "moof".

The actual normative spec, MPEGGroup/isobmff, is only 92 chunks (3.2%). Pure cosine returns the fixtures.

Internal sources (ADO wiki, Confluence) are even smaller — they get buried under sheer volume of spec prose.

Why router wins

Filter before you score

For isobmff-normative: router adds exclude_external_ids = {FileFormatConformance} and pins category = Structural/System. Now cosine runs against ~92 vectors instead of 2,868 — and returns the actual spec.

For ado-wiki-*: router sets sources = {ado_wiki}. The wiki pages stop being needles in a haystack.

Section 4 / 4 · Issues & dilemmas — the honest part VSPP RAG · GenAI Mini-Hackathon

Trade-offs we made — and where they hurt.

Time win is theoretical

At 2,868 chunks, filtering doesn’t save real time.

Cosine over the full index is a single BLAS matmul — sub-millisecond. The Python build_mask we add to “save work” roughly cancels the matmul we skip.

The metadata router’s value here is precision (+31 pts), not latency. The latency win only shows up past ~50 k chunks or when we swap to an ANN index — same code, different scale.

Map-reduce summary is slow on CPU

Local Qwen on a 71 KB spec ≈ minutes.

Single-pass works for ~10 KB docs (Confluence, ADO wiki). Map-reduce on a full 3GPP spec hangs the demo without a GPU.

Mitigation: idempotent via input_sha256 in the manifest — run summary once, never re-run unless content changes. Today only a handful of summaries exist; doc-summary router plumbing is in place but under-populated.

Regex router is brittle

Every new spec family wants a new rule.

The ISOBMFF / DASH / DVB rules in route_query() are hand-coded. Add a new authority → write a new regex → write a new test.

The doc-summary router is exactly the fix — but it only generalizes once enough docs have summaries. We’re half-shipped on that.

No answer step yet

We return passages, not synthesized answers.

Deliberate scope cut for a 4-hour POC: no LLM call at retrieve time → no hallucination, no API cost, no eval-faithfulness rabbit hole. Just citations and let the human read.

Cost: it’s not yet a “product” — a UX layer and an answer-synthesis LLM are the obvious next two days of work.

Section 4 / 4 · Open questions VSPP RAG · GenAI Mini-Hackathon

What we’d explore next — and what we honestly don’t know.

Hybrid retrieval

Spec queries are extremely keyword-dense (“ftyp brand compatibility”, “PAT PMT NIT”). A BM25 leg might out-perform dense for normative terms.

Hypothesis: hybrid (BM25 + MiniLM) gives +1–2 queries on baseline before any router.

Cross-encoder reranker

Top-1 quality matters more than top-5 once a UI exists. A small cross-encoder rerank over the top-20 should fix the “right doc, wrong chunk” cases.

Open: can we keep it CPU-only? Cohere rerank is the obvious cloud fallback.

Answer synthesis

Once we add an LLM at the end, we need a faithfulness metric, not just an external_id match. LLM-as-judge or RAGAS?

Open: which LLM, with which context budget, with which citation format the user can actually click.

Summary coverage

The doc-summary router only shines when ~all 131 docs have summaries. Running Qwen on the full corpus needs a GPU pass or a smaller model.

Open: distill summaries cheaper, or just accept ~30 min on GPU once per spec change.

Scale threshold

At what chunk count does NumPy + masked cosine actually break? Our back-of-envelope says ~50 k chunks before we need ANN.

Open: pgvector vs Qdrant when we cross that line — and how filters become partition keys.

Eval bias

Our 13 queries were written by us. They probably encode our priors about what the router should do well.

Open: mine real engineer questions from Slack / wiki search logs to get an unbiased eval set.

In one line

What makes this not-just-another-RAG

  • Metadata is the index. Every chunk knows its external_id, category, authority, and byte offsets — that’s the routing surface, not an afterthought.
  • Two routers, same filters. Cheap regex for known patterns, doc-summary 2-stage for everything else. One RetrievalFilters object downstream.
  • Receipts on every claim. 9/13 → 13/13 is a measured number from this repo, not a brag.
  • Honest about what doesn’t work yet. CPU-only summaries, brittle regex rules, no answer step. Slide 8 wasn’t marketing.
