4Doc-summary routed retrieval (two-stage)

Enabled with python retrieve.py "..." --doc-summary-router. Stage 1 ranks the tiny set of chunk_kind=doc_summary virtual chunks. Stage 2 reuses the same vector index, restricted to the surviving documents.

Keyword router only

~500–1,500

chunk-vector ops per query (depends on which regex rule fires)

  • Hand-coded rules per spec: DASH, moof, MPD, DVB-S2...
  • Queries not matching any keyword fall back to full scan
  • New specs need new regex rules

Doc-summary router

~180–210

vector ops per query (131 summaries + ~50–80 text chunks)

  • Generalizes to any spec that has a doc_summary
  • One query embedding reused across both stages
  • Fallback to full text search when no summaries match

5Why metadata-first matters

Without metadata filters

2,830

vectors scored per query

  • Conformance fixtures pollute ISOBMFF results
  • Internal wiki leaks into normative-standards answers
  • Slow, no audit trail, no citation guarantees

With metadata-first routing

~50

vectors scored per query (typical)

  • Hard filter: category, source, external_id
  • Internal docs hidden by default unless --include-internal
  • Citable hits: chunk_id + char offsets prove the source
